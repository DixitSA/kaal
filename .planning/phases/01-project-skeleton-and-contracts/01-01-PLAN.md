  <action>
Create the foundational typed contracts for the backend under `src/lib/types` and `src/lib/schemas`. Model the birth input, chart primitives, profile response, decision request/response, and other shared API-facing data shapes described in the project brief. Keep the types modular so later phases can extend them without rewriting the public contracts.

In `src/lib/schemas/input.ts` and `src/lib/schemas/output.ts`, create the initial Zod schemas that mirror the shared API contracts. Include `birthInputSchema` plus the request/response schema boundaries that the route handlers will import later. Keep exact-time and time-unknown considerations visible in the types, even if the detailed execution logic lands in later phases.

Freeze a per-endpoint contract matrix in these shared type/schema files for all four routes:
- `/api/chart/compute`: request body, success body, and error envelope
- `/api/profile/generate`: request body, success body, and error envelope
- `/api/decision/evaluate`: request body, success body, and error envelope
- `/api/today/[userId]`: path-param semantics, success body for a stateless placeholder, and error envelope

Be explicit that `/api/today/[userId]` is database-free in v1 and returns a deterministic stateless placeholder shape rather than implying persistence. Also define one shared error response schema or envelope pattern so Wave 2 can use consistent 400 and placeholder/not-implemented responses without inventing new JSON shapes.
